3. Task 1: A Heap-Based Priority Queue

Imagine a bouncer at a club where only one person is allowed to go in and dance at a time. That’s what it’s like to be an operating system for a single-core CPU. This task is about implementing a data structure that a bouncer can use to keep track of who should go next—a priority queue.

A maximum priority queue is a data structure supporting the following operations:

int queue_init(struct queue *queue); (1)

int queue_push(struct queue *queue, int pri); (2)

int queue_pop(struct queue *queue, int *pri_ptr); (3)

int queue_destroy(struct queue *queue); (4)

  1. Initialize the queue before usage. Should be called before any calls to queue_pop, queue_push, or queue_destroy. Return 0 on success and non-zero if an error occurred. You can assume that queue is not NULL.

  2. Insert an element as a priority. Return 0 on success and non-zero if an error occurred. You can assume that queue is not NULL.

  3. Pop the maximum priority out of the queue. The popped priority is stored at the address given as pri_ptr. Returns 0 on success and non-zero if an error occurred. You can assume that neither queue nor pri_ptr is NULL.

  4. Destroy the queue after usage. Return 0 on success and non-zero if an error occurred. You can assume that queue is not NULL.

struct queue is left for you to define.

Note, the above functions merely retain the priority, and no satellite information. This intentionally keeps the assignment simple. Please adhere to this API as your code might be subject to automated testing.

HANDOUT

We handout a simple bouncer implementation, which you will have to modify to use your priority queue.

  1. bouncer.c — a simple bouncer implementation.

  2. Makefile — allows you to use make to compile, test, and debug your code.

  3. gdb.script and input.txt — a sample GDB script and sample input for testing your code.

  4. README.md — contains information on how to use make to compile, test, and debug your code.

  5. bouncer.py — a quick and dirty bouncer test-suite.

BOUNCER INTERFACE

The bouncer interface is very similar to the stack calculator interface from last week’s exercise sessions. The bouncer is a command-line application which listens for the user to enter a number or a command. Numbers are inserted into the queue, and only two commands are supported: the unary p (meaning "print" or "pop") and the nullary e (meaning exit).

Here is a sample session working with the bouncer:

42
43
p
=> 43
p
=> 42
p
!! Queue underflow.
e

Unlike the stack calculator, the bouncer does not prompt the user with >, and instead signifies output with a leading ⇒ or !!. This is done for testing purposes. Please do not change this API.

YOUR TASK

  1. Implement a module queue (files queue.h and queue.c) declaring and defining struct queue and the functions specified above.

    1. You are free to choose the underlying data structure, so long as it can grow and shrink dynamically. We recommend to start with a linked list or dynamic array implementation, where queue_push or queue_pop takes O(n)O(n) time. See Section 17.4 in [CLRS3e] for hints on how to implement a dynamic array, i.e. an array which grows and shrinks on demand.

    2. If you use a dynamic array implementation, it should be fairly straight forward to convert it to a binary heap using the techniques described in Chapter 6 of [CLRS3e] and achieve O(logn)O(logn) time for both queue_push and queue_pop.

  2. Add a queue.o target to the handed out Makefile and modify bouncer.c to use your queue module. Make sure to properly set up the dependency between bouncer and queue.o.

  3. Try to test that your bouncer works well with an (unexpectedly) large number of guests. To help you test your implementation, we hand out a little Python3 test-script called bounce.py. Just run the script in the same directory as your bouncer and see what happens.

  4. Document your solution in a short (1-2 pages) report. You should (at least):

    1. Discuss your choice of data structure, how it dynamically grows and shrinks on demand. If you use realloc(), discuss how you expect it to work. What happens if realloc() fails?

    2. Document the query interface better than we have done above. For instance, which error code does your queue_pop return if there are no elements queue up in the queue? Can a queue be initialized again after queue_destroy has been called once?

    3. Discuss how you tested your solution and assess the quality of your implementation. Make your tests easily reproducible (e.g. using a phony Makefile target), and tell us how to reproduce your test results.

4. Task 2: KUDOS System Calls for Basic I/O

System calls are the mechanism by which user programs can call kernel functions and get OS service. See the System Calls section in the KUDOS documentation for details on how system calls work in KUDOS.

Each system call in KUDOS has a unique number (defined in kudos/proc/syscall.h), and it can use up to three arguments in registers. A KUDOS user makes a system call by using the MIPS instruction syscall, with register a0 containing the number of the desired system call, and a1, a2 and a3 containing arguments to the kernel function.

The syscall instruction generates an exception, called a trap, and execution continues in kernel mode. With interrupts disabled, a jump is performed: program execution continues at a fixed address with code to save the current (user) context, and then (with interrupts enabled again) proceeds by jumping to a function syscall_entry in file kudos/proc/syscall.c which executes the system call itself (shown here). When syscall_entry returns, the return value of the system call is stored in register v0, and control returns to the user program.

uintptr_t syscall_entry(uintptr_t syscall, uintptr_t arg0, uintptr_t arg1, uintptr_t arg2)
{
  // ...
  switch(syscall)
    {
    case SYSCALL_HALT:
      halt_kernel();
      break;
    default:
      KERNEL_PANIC("Unhandled system call\n");
    }

  /* No return code / value if we reach here */
  return 0;
}

The assembly function _syscall (defined in userland/mips32/_syscall.S) acts as a wrapper around the MIPS machine instruction syscall and can be called from userland code. It is used inside userland/lib.c to define a small userland library.

As can be seen in the code above, only one system call, SYSCALL_HALT is implemented. In order to do anything useful with KUDOS, support for basic console I/O is essential. Your task is to implement support for the system call numbers SYSCALL_READ and SYSCALL_WRITE, when called with the filehandles FILEHANDLE_STDIN and FILEHANDLE_STDOUT (numbers 0 and 1 in kernel/proc/syscall.h), respectively.

In terms of the KUDOS userland library (declared in userland/lib.h), this means support for the following userland functions:

int syscall_read(int filehandle, void *buffer, int length);

int syscall_write(int filehandle, void const *buffer, int length);

See the file-system related section of the KUDOS system calls documentation for more information about how the system calls should behave.

HANDOUT

For every assignment we handout a version of KUDOS we want you to use as the basis for your assignment.

YOUR TASK

  1. Handle the system call numbers SYSCALL_READ and SYSCALL_WRITE using a generic character device. Hint: Look at the code inside init_startup_fallback (file kudos/init/common.c) to see how to acquire and use the generic character device (also see kudos/drivers/gcd.h).

  2. Write a small userland program to test your implementation. Implement it in readwrite.c in the userland/ directory (see userland/halt.c for an example). To test the compiled program, copy the it to the KUDOS disk (e.g. using tools/yams-tfs), and start KUDOS with readwrite as the initial userland program (e.g. using tools/yams-term and tools/yams-init). See Using KUDOS for further instructions on how to do this.

  3. Document your solution in a short (1-2 pages) report. You should (at least):

    1. Discuss how your implementation fits within the overall design of KUDOS.

    2. Discuss in-how-far your implementation is "bullet-proof" (see KUDOS system calls documentation for a definition of bullets).

    3. Discuss how you tested your solution and assess the quality of your implementation. Make your tests easily reproducible (e.g. using a phony Makefile target), and tell us how to reproduce your test results.
