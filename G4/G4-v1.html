This is the fourth in a series of 5 G-assignments ("G" for "Godkendelse" and/or "Gruppeopgave") which you must pass in order to be admitted to the exam in OSM. We encourage pair programming, so please form groups of 2-3 students.

In the first task, we ask you to handle KUDOS TLB exceptions. In the second task, you are to implement a system call for dynamic allocation of memory.

1. Before you start

Along with this assignment, we hand out a fresh KUDOS source based on our reference solution to G2. Please use this as your base.

For background knowledge for this assignment, please consult the chapter on virtual memory in the KUDOS documentation.

2. What to submit

Throughout this course, you should always submit entire, working source code tree in a single tar.gz archive. The uploaded archive should contain:

  1. A source code tree which includes your work for both tasks, and programs you used for testing.

  2. A short report where you document your code, discuss different ways to solve the tasks, explain the design decisions (why you preferred one particular way out of several choices), and assess your solution (especially through testing).

We assess the quality of your work be based on your report. Therefore, it is important to mention the most important parts of the C code that you wrote or modified. Pay particular attention to documenting changes to existing KUDOS code. You should also use comments in the C code itself (but use the report for long explanations).

The KUDOS code you hand in should compile without errors with the original setup. As KUDOS uses -Werror option in its Makefile, your code should also compile without warnings.

3. Task 1: TLB exception handling

KUDOS currently does not handle exceptions caused by TLB misses, i.e. when a page cannot be found in the TLB (Translation Lookaside Buffer). Your task is to implement handlers for these exceptions for KUDOS' MIPS32 target (the one we have used throughout the course).

To reiterate, this is what KUDOS currently does with relation to memory:

  • Whenever a userland process reads from or writes to memory, it goes through the TLB to allow for virtual memory.

  • When a new process is spawned and run, its pagetable is filled into the TLB with no regard for what might already be in the TLB.

  • When the KUDOS scheduler decides to switch to another process, the TLB is again forcefully filled with the new process' pagetable.

We still want to use the TLB, but we would like to do away with all forceful fills: Instead of trying to stuff a process' entire pagetable into the TLB whenever it is running, we would like to only insert a page into a TLB when it is needed. KUDOS throws TLB exceptions when a piece of code attempts to access a piece of memory which is not in the TLB. By handling those TLB exceptions, i.e. by inserting the page in question into the TLB, you can do away with forceful TLB filling.

There are three TLB exceptions:

  • EXCEPTION_TLBL: A memory load operation required a page which either had no entry in the TLB or whose entry was marked as invalid.

  • EXCEPTION_TLBS: A memory store operation required a page which either had no entry in the TLB or whose entry was marked as invalid.

  • EXCEPTION_TLBM: A memory store operation required a page whose dirty bit was 0, indicating that the page is not writable.

Both EXCEPTION_TLBL and EXCEPTION_TLBS should be handled in the same way: The page table of the respective thread is searched for a matching page entry. If found, the page is inserted into the TLB, potentially removing existing entries in the process. If not, or if an invalid page is accessed, it is treated as an error. If the exception originates in userland, the process should exit; if it happens in kernel mode, the kernel must panic.

EXCEPTION_TLBM is always an error. If the exception originates in userland, the process should exit; if it happens in kernel mode, the kernel must panic.

To insert a page into the TLB, we recommend using either a random replacement strategy or a FIFO replacement strategy:

  • When using a random replacement strategy for page entries in the TLB, a new page is just inserted into a random spot. It is the easiest to implement: Just call the function tlb_write_random from kudos/vm/mips32/tlb.h.

  • When using a FIFO (First In, First Out) strategy, a new page is instead inserted into whatever spot contains the first inserted page of the current pages, unless there are unused spots, in which case one of them is used. You can use the function tlb_write for this, but you need to maintain various structures as well.

Implement your handlers as the functions tlb_load_exception, tlb_store_exception, and tlb_modified_exception in the file kudos/vm/mips32/tlb.c. Information about the exception is obtained via _tlb_get_exception_state, which fills a tlb_exception_state_t structure.

Modify the Buenos exception handlers, so that TLB exceptions will trigger the handlers implemented by you. You need to handle both user exceptions — in kudos/proc/mips32/exception.c — and kernel exceptions — in kudos/kernel/mips32/exception.c.

Remove all calls to the function tlb_fill; this function should not be necessary any more, now that TLB misses are handled properly.

Hints:

  • It is not enough to just remove tlb_fill. You must also tell KUDOS about the new ASID (Address Space IDentifier).

  • To find a page in your pagetable that matches the failing virtual address, you can use a for loop.

4. Task 2: Dynamic memory allocation

Add a new field heap_end, which should hold the address of free memory mapped to a physical page, to the process control block. User code can manipulate this heap limit, increasing it in order to allocate more memory when required, through the system call syscall_memlimit.

Implement the system call void* syscall_memlimit(void* new_end):

  • Allocate memory. Set the new heap end (the last addressable byte) to the address new_end given as the parameter.

  • Return the new heap end or NULL on error. If new_end is NULL, the system call instead returns the current heap end.

Please note that the functionality vm_unmap to unmap a memory page from memory is left unimplemented in KUDOS. It is therefore not possible to give back allocated memory to the system. A call to syscall_memlimit to decrease the value of heap_end should be considered an error, as the heap can currently only grow, not shrink.

Finally, make one or more userland test programs that use syscall_memlimit to dynamically allocate memory, and write to and read from the allocated memory. We have already added one such test program; see userland/mem0.c in the handout (but you should make tests of your own as well).

Hints:

  • When spawning a new process, you need to initialize its heap pointer. The initial value should be the beginning of the heap. All binary programs compiled from the userland directory are in the standard ELF format, and KUDOS expects them to contain one read-only (RO) segment and one read-write (RW) segment, which it maps to physical pages in setup_new_process. See the elf_info_t struct of kudos/proc/elf.h for the names of the ELF fields.

  • To extend the heap, you need to map physical pages to virtual addresses; see kudos/vm/memory.h for useful functions.

5. Challenge of the Week

"Challenge of the Week" is a completely optional, typically tough, but also enlightening part of an assignment. Good solutions will be called up during the following Friday lectures, and handed a student-friendly prize.

This week’s challenge is to make memory-reusing malloc and free functions using syscall_memlimit:

void *malloc(size_t size);
void free(void *ptr);

The KUDOS userland library ships with working malloc and free functions, but they act only on a pre-allocated array and are limited by the size of that array. Your job is to remove the dependency on that array and instead use syscall_memlimit to extend the size of the heap whenever malloc needs to do so.

However, your malloc must also reuse memory whenever possible! If we did not make that requirement, this would be a valid solution:

void* malloc(size_t size) {
  uint8_t* cur_end = (uint8_t*) syscall_memlimit(NULL);
  uint8_t* new_end = (uint8_t*) syscall_memlimit((void*) (cur_end + size));
  if (new_end == NULL) {
    return NULL;
  }
  else {
    return (void*) cur_end;
  }
}

void free(void* ptr) {
  return;
}

Instead of extending the memory limit every time malloc is called — like what the above code does — we ask you to write a malloc that reuses a previously allocated memory spot if it has since been freed, and if it is of sufficient size. This also means that you need a more advanced free than the one above…​

You can base your solution on the existing free-list algorithm, or you can create your own from scratch.

Your solution will be judged by its correctness, efficiency, and design.
